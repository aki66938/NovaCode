# NovaCode 开发规章（本仓库生命周期）

> 本文件**随仓库提交**，是 NovaCode 单仓库的开发生命周期规范：提交、版本、CI、发布、分支、AI 协同。
> 适用于所有在本仓库工作的人类开发者与 AI 开发者（Codex、Claude Code 及后续接入的模型）。
> 权限与角色（谁是主创/维护者、如何继承、fork 规则）见 [GOVERNANCE.md](GOVERNANCE.md) 与 [MAINTAINERS.md](MAINTAINERS.md)；外部贡献流程见 [CONTRIBUTING.md](CONTRIBUTING.md)。
> **机密不写入本文件**：签名密钥等保存在本地 gitignore 的 `SIGNING-KEYS.local.md`。
> 每次会话开始前，AI 开发者必须先读取本文件与 `CHANGELOG.md`。

---

## 一、版本号规则

格式：`主版本.里程碑版本.功能版本`，例如 `0.1.1`。

| 位数 | 说明 | 触发条件 |
|------|------|----------|
| 第一位 | 架构大变动 / 年度版本 | 仅由主创决定（主创身份以 `MAINTAINERS.md` 为准） |
| 第二位 | 里程碑版本 | 主创认为功能完善、可面向大部分用户时升级 |
| 第三位 | 功能更新 + Bug 修复 | 每次功能新增或问题修复后累加 |

**关键约束**：
- 版本号升级必须经主创确认，AI 开发者不得单方面决定。
- Git tag 为三段式 SemVer：`v0.1.1`，禁止四段式。
- Release tag 版本须与 `CHANGELOG.md` 对应版本一致；`release.yml` 按 tag 自动 patch `tauri.conf.json` 的 version。
- 第三位持续累加，由主创判断是否值得单独发版。

## 二、分支与 PR 流程（0.1.x 起强制）

```
fork / feat·fix 分支  ──PR──▶  dev  ──主创真机验证──▶  main  ──tag vX.Y.Z──▶  Release
```

- **`main`**：稳定发布线。受保护，**禁止直接 push**，只接受来自 `dev` 的合并；克隆者/用户看到的就是它。每次 `dev → main` 合并对应一次发版。
- **`dev`**：集成与验证线。所有特性、修复先在此汇合。
- **流程**：
  1. 基于 `dev` 建 `feat/<主题>` 或 `fix/<主题>`（外部贡献者先 fork）。
  2. 开发 + 自测：`cargo test --workspace` 与 `npm run typecheck` 通过。
  3. 人类发起 PR，**base 选 `dev`**；`build.yml` 对 PR 跑编译验证。
  4. 维护者 review 后合并进 `dev`。
  5. **主创在 `dev` 上做真机 app 体验验证**（`npx tauri dev` 或安装包），确认功能可用。
  6. 主创合并 `dev → main`，打三段式 tag，触发 `release.yml` 出安装包。
- **谁能合并 `dev → main`、谁能打 tag**：由 `MAINTAINERS.md` 定义。
- **紧急修复**：可从 `main` 拉 `fix/<主题>` 直接 PR 回 `main`，发版后回灌 `dev`；此为例外，需主创批准。

## 三、提交前验证规则（强制）

任何涉及代码的改动，必须本地验证通过后才能并入：

1. AI 自测：`cargo test --workspace` + `npm run typecheck`（或 `npx tsc --noEmit`）全部通过。
2. 在终端运行 `npx tauri dev`（`apps/desktop` 目录），确认正常启动、无编译错误、无运行时崩溃。
3. 人工走通改动涉及的功能路径。
4. 主创/维护者确认「跑通了」后，才合并。

AI 开发者完成修改后先跑通测试与类型检查，再请人类验证，**不得在人类未确认时自行合并到 `dev`/`main` 或打 tag**。

## 四、Commit 规则

```
<type>(<scope>): <summary>

[可选 body]

Co-Authored-By: <AI 模型名> <邮箱>
```

type：`feat` `fix` `refactor` `style` `docs` `chore` `test`。
scope 示例：`ui`、`deepseek-client`、`mcp`、`token-accounting`、`storage`、`sandbox`、`ci`、`agent`。

**强制**：
- summary 祈使句、客观、≤ 80 字符。
- 一次提交只做一件事，禁止混入无关改动。
- 文件统一 **UTF-8（无 BOM）**；提交前确认中文无乱码。
- 提交以 AI 开发者署名结尾（`Co-Authored-By:`），AI 必须如实声明自己是哪个模型，不得冒充。

**禁止**：空 message；超 80 字符 summary；提交明文密钥；`git push --force` 到 `main`/`dev`；`--no-verify`/绕过签名（除非主创明确要求）；提交 `target/`、`node_modules/`、`dist/`、`*.db`、`*.local.md`、本地密钥。

## 五、CI/CD 触发规则

### Release（`release.yml`）
- **触发**：推送 `v*.*.*` tag。
- **产出**：Windows `.msi` + `.exe` + `latest.json` + `.sig`，上传到 GitHub Release。
- **何时必须打 tag**：`dev → main` 合并后、主创已真机验证、且涉及 `apps/**`、`crates/**`、`packages/**`、`Cargo.*`、`package*.json`、`tauri.conf.json` 的源码/依赖/构建变更，默认必须打新三段式 tag。
- **不打 tag**：仅改 `CHANGELOG.md`/`README.md`/治理文档/静态资源；仅注释/格式化/无功能 refactor；仅改 CI 配置无功能变化；主创认为不足以发版。

### Build（`build.yml`）
- **触发**：push 或 PR 到 `main`/`dev`，且 `apps/**`、`crates/**`、`packages/**`、`Cargo.*`、`package*.json` 有变动。
- **产出**：仅验证编译，不出包。

### 迭代节奏与版本归属（主创裁决，优先级最高）
- **CI 期间不停工**：推送 tag 后 CI 约 15-30 分钟，期间继续下一批编码，不等待结果。
- **版本归属**：
  - CI **构建失败** → 修复与新增**并入同一版本号**；修复经主创验证后，删除远端旧 tag、把同名 tag 重打到新 commit 重新触发。
  - CI **成功出包** → 新改动用**下一个版本号**单独发版。
- AI 不得自决版本归属；主创告知结果前，新改动只编码、自测，不合并/打 tag。

## 七、CHANGELOG 更新规则

每次发布前在 `CHANGELOG.md` 表格**新增一行**：版本号 | 面向用户的更新描述（`<br>` 分隔多条） | 开发者署名。
- 一个版本号一行；描述面向最终用户，不写实现细节；Release 说明直接复用该行。

---

## 八、AI 开发者协同原则

- 会话开始先读 `.dev-rules.md` 与 `CHANGELOG.md`。
- 不重复其他开发者（人或 AI）在途内容；不单方面改版本号。
- **最小变更**：贴合既有风格、命名、注释密度，不顺手重构无关代码。
- 沙箱与权限 **fail-closed**：绝不伪造成功、绝不静默降级。
- 命令子进程必须擦除密钥类环境变量（`*_API_KEY`、`*SECRET*`、`*TOKEN*`、`*PASSWORD*`）。

---

## 九、AI 写作与文档规范

- **准确第一**：只写代码确实实现的能力；未实现/未验证功能必须显式标注，严禁夸大臆造。
- **身份一致**：本应用是 NovaCode。配置目录 `.novacode/`，记忆文件 `NovaCode.md`，数据库 `novacode.db`。严禁在代码、提示词或文档中引入 `.claude/` 等其他工具的目录与约定。
- **编码与语言**：面向中文用户的文档以中文为主，slogan 中英双语；统一 UTF-8（无 BOM），核对不乱码。
- **出处署名（许可证一致）**：NovaCode 采用 Modified MIT，衍生作品须保留 `Based on NovaCode by aki66938`；不得移除或误述出处。
- **面向用户而非实现**：CHANGELOG/Release 写用户可感知的变化。
- **不臆造外部事实**：第三方模型/接口的能力与价格以可核实来源为准。

## 十、供应商演进约束（架构方向）

- NovaCode 定位为面向多家国产大模型的通用客户端；**多供应商架构为后续独立计划**，不在当前迭代擅自展开。
- 保持现有 DeepSeek 客户端可用；产品文案不得写成「仅服务于 DeepSeek」。
- 引入新供应商前先与主创确认接口、适配层、设置 UI、计价与令牌账本、模型能力元数据设计。
