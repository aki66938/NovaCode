# NovaCode 治理规章

> 本文件定义 NovaCode 项目的角色、权力、决策与继承规则。它面向所有参与者——人类与 AI——以及未来 fork 二次开发的主创（可作为模板自行优化）。
> 仓库的开发生命周期（提交/版本/CI/分支细则）见 [.dev-rules.md](.dev-rules.md)；当前权限名单见 [MAINTAINERS.md](MAINTAINERS.md)；贡献流程见 [CONTRIBUTING.md](CONTRIBUTING.md)。

## 1. 角色与权力

| 角色 | 权力 | 边界 |
|------|------|------|
| **主创（Lead）** | 版本号、tag、发版、架构方向、治理规则、维护者任免的最终裁决权 | 见 `MAINTAINERS.md`，当前为 [@aki66938](https://github.com/aki66938) |
| **维护者（Maintainer）** | review、合并 PR 进 `dev` | 不得单独改版本线或执行 `dev → main` 发版（除非主创授权） |
| **贡献者（Contributor）** | 提交 PR、参与讨论 | 须遵守 `CONTRIBUTING.md` 与许可证署名 |
| **AI 开发者** | 在人类指挥下编码、自测、起草 | 永不自决版本/tag/发版；须经人类「可以提交」闸门；须如实声明自身模型身份 |

## 2. 决策原则

- **人类裁决、AI 执行**：AI 提出方案与实现，版本/发布/架构等决策由人类（主创）拍板。
- **权力事实源单一**：谁有权由 `MAINTAINERS.md` 唯一定义，其他文档只引用、不重复定义。
- **最小惊讶**：影响外部用户或不可逆的操作（发版、移动 tag、改许可证、改默认分支）须主创批准。
- **透明记账**：版本归属、tag 移动等以提交与 `CHANGELOG.md` 记录在案。

## 3. 分支与发布权

- `main` 受保护，仅主创可执行 `dev → main` 合并与打 tag 发版。
- `dev` 为集成线，维护者可合并经审阅的 PR。
- 详细流程见 `.dev-rules.md`「分支与 PR 流程」。

## 4. 主创继承与转移

- **主动转让/增设**：现任主创通过修改 `MAINTAINERS.md` 的 PR 完成，并在描述中交接。
- **长期缺席**：若主创连续 6 个月无响应且未指定继任，由现任维护者协商推举新主创，并以更新 `MAINTAINERS.md` 的提交确立；无维护者时，项目可由社区 fork 延续（见下）。
- 继任者继承全部主创权力与对应责任（含密钥与发布渠道的重新建立）。

## 5. Fork 与独立分支

- 任何人可 fork 本仓库进行二次开发。fork 后即为其自有项目的主创，自定治理。
- 唯一硬约束：遵守 [LICENSE](LICENSE)（Modified MIT）的**出处署名**——在文档或「关于」等显著位置保留
  `Based on NovaCode by aki66938 (https://github.com/aki66938/NovaCode)`；大规模商用须显著展示 "NovaCode"。
- 本治理文档对独立 fork 仅为**参考标准**，fork 主创可自行优化。

## 6. AI 开发者准则（治理层）

- 每个 AI 开发者（Codex、Claude Code 及后续模型）在 commit 中以 `Co-Authored-By` 如实声明身份，不得冒充其他模型或工具。
- AI 不得引入非本项目的目录/约定（如 `.claude/`），不得在文档中夸大或臆造未实现/未验证的能力。
- AI 的工作分配由主创临时指挥，不设固定分工。
